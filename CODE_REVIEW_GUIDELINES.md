# 电商AI Agents代码审查指南

## 🎯 审查目标

### 核心目标
1. **确保代码质量** - 代码符合项目标准
2. **防止安全漏洞** - 代码没有安全风险
3. **保持一致性** - 代码风格和架构一致
4. **促进学习** - 通过审查提高团队技能

### 审查原则
- 🤝 **建设性反馈** - 提供有帮助的建议
- ⏱️ **及时响应** - 在合理时间内完成审查
- 📚 **教育导向** - 帮助贡献者学习和改进
- 🔍 **全面检查** - 检查所有重要方面

## 📋 审查清单

### 1. 安全性审查
#### 必须检查的项目：
- [ ] **输入验证**: 所有用户输入都经过验证
- [ ] **SQL注入**: 没有SQL注入风险
- [ ] **XSS攻击**: 防止跨站脚本攻击
- [ ] **权限检查**: 适当的权限验证
- [ ] **敏感信息**: 没有硬编码的敏感信息
- [ ] **依赖安全**: 依赖包没有已知漏洞

#### 安全红线（直接拒绝）：
- ❌ 硬编码密码或密钥
- ❌ 明显的安全漏洞
- ❌ 绕过权限检查的代码
- ❌ 可能造成数据泄露的代码

### 2. 代码质量审查
#### 代码规范：
- [ ] **命名规范**: 变量、函数命名清晰
- [ ] **代码结构**: 逻辑清晰，结构合理
- [ ] **注释质量**: 必要的注释，解释复杂逻辑
- [ ] **函数长度**: 函数不超过50行（理想情况）
- [ ] **复杂度**: 代码复杂度合理

#### 质量标准：
- ✅ **可读性**: 代码易于理解
- ✅ **可维护性**: 易于修改和扩展
- ✅ **可测试性**: 易于编写测试
- ✅ **性能**: 没有明显的性能问题

### 3. 功能审查
#### 功能正确性：
- [ ] **需求符合**: 代码实现需求功能
- [ ] **边界情况**: 处理了边界情况
- [ ] **错误处理**: 适当的错误处理机制
- [ ] **用户体验**: 提供良好的用户体验

#### 兼容性检查：
- ✅ **向后兼容**: 不影响现有功能
- ✅ **API兼容**: API变更有相应文档
- ✅ **浏览器兼容**: 前端代码浏览器兼容
- ✅ **移动端兼容**: 响应式设计检查

### 4. 测试审查
#### 测试要求：
- [ ] **单元测试**: 有相应的单元测试
- [ ] **测试覆盖**: 关键逻辑有测试覆盖
- [ ] **测试质量**: 测试用例设计合理
- [ ] **集成测试**: 影响多个模块时有集成测试

#### 测试标准：
- ✅ **测试通过**: 所有测试通过
- ✅ **测试独立**: 测试不依赖外部环境
- ✅ **测试速度**: 测试运行速度合理
- ✅ **测试文档**: 测试有必要的说明

### 5. 文档审查
#### 文档要求：
- [ ] **代码注释**: 关键逻辑有注释
- [ ] **API文档**: API有相应的文档
- [ ] **使用示例**: 提供使用示例
- [ ] **变更记录**: 记录重要变更

#### 文档标准：
- ✅ **准确性**: 文档内容准确
- ✅ **完整性**: 文档覆盖所有重要点
- ✅ **可读性**: 文档易于理解
- ✅ **及时性**: 文档与代码同步更新

## 🔄 审查流程

### 1. 提交阶段
```
贡献者提交PR → 自动检查 → 等待人工审查
```

#### 自动检查项目：
- ✅ 代码格式检查 (Prettier/ESLint)
- ✅ 单元测试运行
- ✅ 安全扫描 (依赖漏洞检查)
- ✅ 构建检查 (能否成功构建)

### 2. 人工审查阶段
#### 第一轮审查（快速审查）：
- ⏱️ **时间**: 24小时内响应
- 🔍 **重点**: 安全性、基本质量
- 📝 **反馈**: 提供初步反馈

#### 第二轮审查（详细审查）：
- ⏱️ **时间**: 3天内完成
- 🔍 **重点**: 功能、测试、文档
- 📝 **反馈**: 详细审查意见

#### 第三轮审查（最终审查）：
- ⏱️ **时间**: 1天内完成
- 🔍 **重点**: 修改后的代码
- 📝 **决定**: 批准或继续修改

### 3. 批准与合并
#### 批准条件：
- ✅ 所有自动检查通过
- ✅ 至少2名审查者批准
- ✅ 没有未解决的安全问题
- ✅ 测试覆盖率达标

#### 合并流程：
1. 审查者批准PR
2. 贡献者解决所有评论
3. 维护者进行最终检查
4. 合并到主分支
5. 触发CI/CD流程

## 💬 审查沟通

### 1. 沟通原则
- 🗣️ **尊重礼貌**: 使用礼貌的语言
- 💡 **具体明确**: 提供具体的改进建议
- ❓ **提问引导**: 通过提问引导思考
- 👍 **肯定优点**: 先肯定代码的优点

### 2. 沟通模板
#### 正面反馈：
```
👍 这部分代码写得很好，特别是[具体优点]。
```

#### 改进建议：
```
💡 建议考虑[具体建议]，因为[原因]。
```

#### 问题指出：
```
❓ 这里[具体代码]可能会遇到[具体问题]，考虑过如何处理吗？
```

#### 安全提醒：
```
🔒 这里需要注意[安全问题]，建议[解决方案]。
```

### 3. 争议处理
#### 处理原则：
1. **技术优先**: 基于技术事实讨论
2. **数据驱动**: 提供测试数据支持
3. **多方意见**: 征求其他审查者意见
4. **维护者裁决**: 最后由维护者决定

#### 争议解决流程：
1. 贡献者与审查者讨论
2. 征求第三方意见
3. 技术委员会讨论
4. 维护者最终决定
